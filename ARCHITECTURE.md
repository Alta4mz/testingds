# Обоснование архитектурных решений

## 1. Выбор коллекций

### List для хранения компонентов/команд/результатов
- **List<Car> cars в Team** — порядок болидов не критичен, но List проще для итерации и добавления. Map был бы избыточен, т.к. поиск по ID не требуется.
- **List<RaceResult> playerRaceHistory** — важен порядок добавления (хронология гонок), поэтому List, а не Set.
- **List<Component> marketComponents** — компоненты на рынке могут дублироваться (несколько одинаковых двигателей), поэтому не Set.
- **List<TrackSegmentType> segments в Track** — критически важен порядок сегментов трассы для расчёта сложности.

### Почему не Map?
Map использовался бы, если бы требовался быстрый поиск по ключу (например, по ID компонента). В ТЗ такой потребности нет, а List проще в реализации и чтении.

## 2. Принципы ООП

### Инкапсуляция
- **Все поля private** во всех модельных классах (Component, Car, Pilot, Team и др.)
- Доступ только через геттеры/сеттеры с валидацией (например, `setWear()` в Component проверяет диапазон [0,1])
- Пример: класс `Component` скрывает поля `wear` и `isBroken`, предоставляя методы `addWear()`, `repair()`, `isBroken()`

### Наследование
- **Component → Engine, Chassis, Transmission, Suspension, AeroKit, Tire** — общая логика износа и производительности вынесена в базовый класс
- **Staff → Engineer, Mechanic** — общий интерфейс сотрудника с абстрактным методом `getPerformanceBonus()`, который переопределяется в наследниках
- Пример полиморфизма: `team.getTotalEngineerBonus()` вызывает `getPerformanceBonus()` для каждого Staff, но фактически работает только с Engineer (проверка instanceof)

### Полиморфизм
- **Comparable<RaceResult>** — результат гонки сортируется через `compareTo()`, что позволяет использовать `Collections.sort()`
- **instanceof в Team.hasEngineer()** — проверка типа сотрудника без явного приведения к конкретному классу в UI

### Абстракция
- **Abstract class Component** — определяет общий контракт для всех компонентов (износ, ремонт, производительность)
- **Abstract class Staff** — определяет интерфейс сотрудника (роль, квалификация, бонус)
- **Enum ComponentType, WeatherCondition, TrackSegmentType** — типобезопасные перечисления вместо magic strings

## 3. Отсутствие дублирования кода

### Вынос общей логики
- **ValidationUtil** — все проверки «здравого смысла» (неотрицательность, диапазон [0,1], достаточность бюджета) вынесены в утилитный класс
- **CompatibilityChecker** — логика проверки совместимости компонентов инкапсулирована в отдельном сервисе, а не размазана по UI
- **RaceCalculator** — формула расчёта времени круга в одном месте, что позволяет легко менять баланс

### Разделение UI и логики
- **GameMenu** — только ввод/вывод и маршрутизация команд
- **Сервисы (RaceCalculator, BotTeamGenerator, CompatibilityChecker)** — бизнес-логика без зависимостей на консоль
- **Модели (Car, Team, Component)** — только данные и простые методы доступа

Пример: метод `simulateRace()` в GameMenu вызывает `RaceCalculator.calculateLapTime()`, но не содержит формул расчёта.

## 4. Работа Debug-режима

### Класс DebugInitializer
- **createDebugGame()** — фабричный метод, создающий полностью готовую конфигурацию:
  - Бюджет 10_000_000
  - Болид со всеми 6 компонентами (двигатель, шасси, трансмиссия, подвеска, аэропакет, шины)
  - Пилот с высокими статами (accuracy=0.85, aggression=0.75, experience=0.9)
  - Инженер квалификации 10 (бонус 5 сек к времени круга)
  - 3 команды ботов со случайными, но валидными конфигурациями

- **main()** — точка входа для запуска игры в 1 клик без необходимости проходить меню покупки/сборки

### Преимущества
- Преподаватель может запустить гонку сразу после компиляции
- Не нужно тратить время на покупку компонентов и найм персонала
- Демонстрирует работоспособность всех систем (совместимость, расчёт времени, износ, инциденты)

## 5. Маленькие методы

### Принцип 1 метод = 1 действие
- **Car.getEffectiveSpeed()** — только расчёт скорости, без побочных эффектов
- **Car.hasHighWearComponents()** — только проверка, без вывода на консоль
- **RaceCalculator.calculateLapTime()** — разбит на подфакторы (weatherFactor, pilotFactor, engineerBonus)
- **GameMenu.readIntChoice()** — только валидация ввода, без бизнес-логики

Все методы короче 30 строк. Сложная логика разбита на приватные вспомогательные методы.

## 6. Валидация «здравого смысла»

### Реализованные проверки
1. **Нельзя купить без денег** — `Team.spendBudget()` возвращает false, если бюджет недостаточен
2. **Нельзя поставить 2 двигателя** — структура Car имеет ровно одно поле `engine`, повторная установка перезаписывает старое
3. **Нельзя запустить гонку без болида/пилота/инженера** — `startRaceMenu()` проверяет `getCars().isEmpty()`, `getPilots().isEmpty()`, `hasEngineer()`
4. **Нельзя нанять дубликат** — в текущей реализации не запрещено явно, но можно добавить проверку по имени в `Team.addPilot()`/`addStaff()`

### Валидация в конструкторах
- Все конструкторы моделей вызывают `ValidationUtil.validate*()` для входных параметров
- Пример: `new Pilot()` выбросит IllegalArgumentException, если accuracy вне [0,1]

## 7. Формула расчёта времени круга

### Явное использование 5 факторов из ТЗ
```java
baseTime = trackLength / car.getEffectiveSpeed()
finalTime = baseTime * weatherFactor * pilotFactor * trackDifficulty - engineerBonus
```

1. **Характеристики Болида** — `car.getEffectiveSpeed()` учитывает все 6 компонентов
2. **Действия персонала** — бонус механика мог быть добавлен через `Mechanic.getRepairEfficiency()`
3. **Квалификация инженеров** — `team.getTotalEngineerBonus()` = сумма `qualification * 0.5` для всех инженеров
4. **Особенности Пилота** — `pilot.getPilotTimeModifier()` = `1 + aggression*0.05 - accuracy*0.03`
5. **Особенности трассы** — `track.getTrackDifficultyMultiplier()` зависит от количества поворотов/прямых

### Погода
- `WeatherCondition` enum с коэффициентом: DRY=1.0, CLOUDY=1.05, RAIN=1.25, HEAVY_RAIN=1.4

## 8. Альтернативные решения

### Хранение компонентов в инвентаре
- **Текущее решение**: `List<Car> componentInventory` — каждый Car хранит один компонент
- **Альтернатива**: `Map<ComponentType, List<Component>>` — более явно, но сложнее в реализации
- **Обоснование**: для лабораторной работы достаточно простого решения, Map добавил бы 20+ строк кода без существенного усложнения логики

### Проверка совместимости
- **Текущее решение**: возврат `List<String>` с ошибками
- **Альтернатива**: возврат boolean + вывод ошибок сразу в UI
- **Обоснование**: разделение логики проверки и отображения соответствует SRP (Single Responsibility Principle)

## 9. Структура пакетов

```
com.zovauto
├── model          # Сущности (данные, состояние)
│   ├── Component  # Абстрактный базовый класс
│   ├── Engine, Chassis, ...  # Конкретные компоненты
│   ├── Car        # Агрегат компонентов
│   ├── Pilot      # Пилот
│   ├── Staff      # Абстрактный сотрудник
│   ├── Engineer, Mechanic    # Конкретные роли
│   ├── Team       # Команда (игрок или бот)
│   ├── Track      # Трасса
│   └── RaceResult # Результат гонки
├── service        # Бизнес-логика
│   ├── CompatibilityChecker  # Проверка совместимости
│   ├── RaceCalculator        # Расчёт времени круга
│   └── BotTeamGenerator      # Генерация ботов
├── ui             # Консольное меню
│   └── GameMenu   # Главное меню и обработка ввода
├── util           # Утилиты
│   ├── ComponentType, WeatherCondition, TrackSegmentType  # Enums
│   └── ValidationUtil         # Валидация данных
└── DebugInitializer  # Точка входа для debug-режима
```

Такая структура соответствует принципу разделения ответственности и облегчает навигацию по коду.
